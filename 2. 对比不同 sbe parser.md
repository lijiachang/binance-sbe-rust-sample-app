## 核心机制对比

### 读取单个 i64 字段的实际执行路径

**方案 A** (`spot_stream` — SbeTool 生成)：

```278:280:spot_stream/src/best_bid_ask_stream_event_codec.rs
        pub fn event_time(&self) -> i64 {
            self.get_buf().get_i64_at(self.offset)
        }
```

底层调用链：`get_i64_at` → `get_bytes_at` → `slice[index..index+8].try_into()` → `from_le_bytes`。**一次** slice range 边界检查，直接返回 `i64`，无 `Result` 包装。

**方案 B** (`nautilus_trader` — 手写)：

```83:88:nautilus_trader/crates/adapters/binance/src/common/sbe/stream/best_bid_ask.rs
        let event_time_us = cursor.read_i64_le()?;
        let book_update_id = cursor.read_i64_le()?;
        let price_exponent = cursor.read_i8()?;
        let qty_exponent = cursor.read_i8()?;
```

每个 `read_i64_le()` 的底层：

```220:233:nautilus_trader/crates/serialization/src/sbe/cursor.rs
    pub fn read_i64_le(&mut self) -> Result<i64, SbeDecodeError> {
        self.require(8)?;
        let value = i64::from_le_bytes([
            self.buf[self.pos],
            self.buf[self.pos + 1],
            self.buf[self.pos + 2],
            self.buf[self.pos + 3],
            self.buf[self.pos + 4],
            self.buf[self.pos + 5],
            self.buf[self.pos + 6],
            self.buf[self.pos + 7],
        ]);
        self.pos += 8;
        Ok(value)
    }
```

`require(8)` 检查 + **8 次**独立的数组索引 + `Result` 包装 + `?` 解包。

## 逐项对比

### 1. 边界检查开销

| | 方案 A | 方案 B |
|---|---|---|
| 读 1 个 i64 | 1 次 slice range check | 1 次 `require` + 8 次 `buf[pos+i]` 索引 |
| 读 BestBidAsk 全部字段 | ~8 次 range check | 8 次 `require` + 50+ 次数组索引 |
| LLVM 优化可能性 | range check 已经证明了区间，内部无额外 check | 编译器*可能*推导出 `require` 后索引安全，但 8 个独立索引更难优化 |

方案 A 更有利于 LLVM 消除冗余边界检查。

### 2. 零拷贝程度

| | 方案 A | 方案 B |
|---|---|---|
| 固定字段 | 直接从 `&[u8]` 读取，零拷贝 | 同样从 `&[u8]` 读取，零拷贝 |
| symbol 字段 | 返回 `(offset, len)` 坐标 + `&[u8]`，**完全零拷贝** | `read_var_string8_ref` 返回 `&str`（零拷贝），但随后 `Ustr::from()` 做了**全局字符串驻留** |
| 整体 | 不分配任何内存 | 分配 `BestBidAskStreamEvent` 结构体（栈上，便宜） + `Ustr` 驻留开销 |

方案 A 的 symbol 处理是彻底的零拷贝；方案 B 的 `Ustr::from()` 涉及全局 hash table 查找（有锁/原子操作）。

### 3. 按需读取 vs 一次性解码

**方案 A 是懒读取**：调用 `.bid_price()` 时才从 buffer 中读取那个字段。如果你只关心 `bidPrice` 和 `askPrice`，其他字段的读取开销为零。

**方案 B 是 eager 一次性解码**：调用 `decode()` 时所有字段都会被读取并填入结构体。即使你只需要 `bidPrice`，`eventTime`、`bookUpdateId`、`symbol` 等都会被解码。

对于高频场景中"只看价格不看其他字段"的需求，方案 A 有优势。

### 4. Repeating Group（Depth 消息的关键差异）

这是差距最大的地方。以 `DepthSnapshotStreamEvent`（20 档深度）为例：

**方案 A**：

```498:508:spot_stream/src/depth_snapshot_stream_event_codec.rs

```

零分配的迭代器模式，每次 `advance()` 只移动 offset，字段读取直接走 `get_i64_at`。**无 Vec 分配**。

**方案 B**：

```84:88:nautilus_trader/crates/adapters/binance/src/common/sbe/stream/depth_snapshot.rs
        let bids = cursor.read_group(bid_block_length, u32::from(num_bids), PriceLevel::decode)?;
// read_group 内部：Vec::with_capacity(count) + 逐个 decode 填入
```

```396:408:nautilus_trader/crates/serialization/src/sbe/cursor.rs
        let mut items = Vec::with_capacity(count);
        for _ in 0..count {
            let item_start = self.pos;
            let item = decode_item(self)?;
            items.push(item);
            self.pos = item_start + block_len;

```

每次 decode 都会 `Vec::with_capacity(20)` 分配堆内存。对于 20 档 × 2（bids + asks）= 2 次堆分配，每秒数千条 depth 消息时这个开销是可观的。

### 5. 安全性与健壮性

| | 方案 A | 方案 B |
|---|---|---|
| 越界处理 | **panic**（`expect("slice with incorrect length")`） | **Result 返回错误**，不 panic |
| 恶意 payload | 无 group size 上限保护 | `MAX_GROUP_SIZE = 10_000` 防止 OOM |
| schema 校验 | `debug_assert` 仅在 debug 模式 | 运行时严格校验 schema_id |
| UTF-8 校验 | 返回 `&[u8]`，不校验 | `str::from_utf8` 校验 |

**方案 B 在生产环境中更安全**，面对网络上的畸形数据不会 panic。

## 性能估算（BestBidAskStreamEvent 单条解码）

在 release 模式 `-O2` 下，单条 BestBidAsk（约 60 字节）：

| | 方案 A | 方案 B |
|---|---|---|
| 预计耗时 | ~5-10 ns | ~15-30 ns |
| 主要开销 | 几乎全是内存读取 | `Result` 分支 + `Ustr` 驻留 |
| 堆分配 | 0 | 0（首次 symbol 除外） |

对于 `DepthSnapshot`（20 档）：

| | 方案 A | 方案 B |
|---|---|---|
| 预计耗时 | ~30-50 ns | ~80-150 ns |
| 主要差距来源 | 零分配迭代 | 2 次 `Vec::with_capacity` 堆分配 |

## 总结

| 维度 | 方案 A（SbeTool 生成） | 方案 B（Nautilus 手写） |
|------|----------------------|----------------------|
| **纯解码速度** | 更快（~2-3x） | 稍慢 |
| **内存分配** | 零分配 | 有堆分配（Vec, Ustr） |
| **安全性** | panic on 畸形数据 | Result 优雅降级 |
| **按需读取** | 支持（懒读取） | 不支持（全量解码） |
| **易用性** | API 较原始（坐标式 symbol） | API 友好（直接拿结构体） |
| **可维护性** | schema 更新后重新生成即可 | 需要手动同步 |

**如果追求极致性能**（HFT 场景）：方案 A 更合适，零分配 + 懒读取 + 更少的分支。

**如果追求生产稳健性**：方案 B 的 `Result` 错误处理和 `MAX_GROUP_SIZE` 防护更适合面对不可信的网络数据。

**实际上最好的方案**是：用方案 A 的生成代码做底层 codec，在上层包一层 Result 错误处理（类似方案 B 的 `StreamDecodeError`），兼得两者的优点。
