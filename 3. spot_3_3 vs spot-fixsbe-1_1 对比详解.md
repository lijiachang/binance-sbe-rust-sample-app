# spot_3_3.xml vs spot-fixsbe-1_1.xml 对比详解

> 笔记日期：2026-03-11
>
> 核心结论：这两套 schema 分别服务于 **完全不同的接入协议**——`spot_3_3.xml` 用于 REST/WebSocket API，`spot-fixsbe-1_1.xml` 用于 FIX 协议。
> 它们的 schema ID、消息头格式、字段命名风格、枚举编码方式和适用场景都不同，**不能混用**。

---

## 一、总览对比

| 维度 | `spot_3_3.xml` | `spot-fixsbe-1_1.xml` |
|------|---------------|----------------------|
| **所属协议** | REST API / WebSocket API | FIX 协议（TCP+TLS） |
| **Package 名** | `spot_sbe` | `fix_sbe` |
| **Schema ID** | `3` | `1` |
| **当前版本** | `3:3`（2026-03-09 滚动部署） | `1:1`（2026-03-09 滚动部署） |
| **Latest 别名** | `spot_prod_latest.xml` | `spot_fix_prod_latest.xml` |
| **字节序** | 小端（littleEndian） | 小端（littleEndian） |
| **Semantic Version** | `5.2` | `5.2` |
| **消息数量** | **100+ 种** | **约 30 种** |
| **目标用户** | 一般开发者、量化交易者 | 机构/专业交易者（FIX 基础设施） |

## 二、接入方式对比

### 2.1 `spot_3_3.xml` — REST / WebSocket API

**REST API 使用方式：**

```bash
curl -X GET \
  -H 'Accept: application/sbe' \
  -H 'X-MBX-SBE: 3:3' \
  'https://api.binance.com/api/v3/exchangeInfo?symbol=BTCUSDT'
```

**WebSocket API 使用方式：**

```
wss://ws-api.binance.com:443/ws-api/v3?responseFormat=sbe&sbeSchemaId=3&sbeSchemaVersion=3
```

- 请求使用 JSON（REST 的 HTTP 参数或 WS API 的 JSON text frame）
- 响应使用 SBE 二进制编码
- 认证方式：API Key + HMAC/RSA/Ed25519 签名

### 2.2 `spot-fixsbe-1_1.xml` — FIX 协议

**端点：**

| 会话类型 | FIX 文本请求 + SBE 响应 | FIX SBE 请求 + SBE 响应 |
|---------|---------------------|---------------------|
| Order Entry | `fix-oe.binance.com:9001` | `fix-oe.binance.com:9002` |
| Drop Copy | `fix-dc.binance.com:9001` | `fix-dc.binance.com:9002` |
| Market Data | `fix-md.binance.com:9001` | `fix-md.binance.com:9002` |

- 需要 TCP+TLS 连接（不是 HTTP/WebSocket）
- 必须使用 **Ed25519** API Key（不支持 HMAC/RSA）
- 需要 FIX 会话管理：Logon → Heartbeat → Logout
- API Key 需要 `FIX_API` 或 `FIX_API_READ_ONLY` 权限

## 三、消息头（Message Header）对比

这是两套 schema 最根本的结构性差异之一。

### 3.1 `spot_3_3.xml` 消息头

标准 SBE 消息头，8 字节：

```xml
<composite name="messageHeader">
    <type name="blockLength" primitiveType="uint16"/>  <!-- 2 字节 -->
    <type name="templateId"  primitiveType="uint16"/>  <!-- 2 字节 -->
    <type name="schemaId"    primitiveType="uint16"/>  <!-- 2 字节 -->
    <type name="version"     primitiveType="uint16"/>  <!-- 2 字节 -->
</composite>
```

**总大小：8 字节**，无 SOFH，无序列号，无发送时间。

### 3.2 `spot-fixsbe-1_1.xml` 消息头

FIX SBE 消息头包含 SOFH + 扩展 header，共 **26 字节**：

```xml
<!-- SOFH: Simple Open Framing Header, 6 字节 -->
<composite name="sofh">
    <type name="messageLength" primitiveType="uint32"/>  <!-- 4 字节：包含 SOFH 自身的总长度 -->
    <type name="encodingType"  primitiveType="uint16"/>  <!-- 2 字节：必须为 0xEB50（小端） -->
</composite>

<!-- Message Header: 20 字节 -->
<composite name="messageHeader">
    <type name="blockLength" primitiveType="uint16"/>  <!-- 2 字节 -->
    <type name="templateId"  primitiveType="uint16"/>  <!-- 2 字节 -->
    <type name="schemaId"    primitiveType="uint16"/>  <!-- 2 字节 -->
    <type name="version"     primitiveType="uint16"/>  <!-- 2 字节 -->
    <type name="seqNum"      primitiveType="uint32"/>  <!-- 4 字节：消息序列号 -->
    <ref  name="sendingTime" type="utcTimestampUs"/>    <!-- 8 字节：发送时间戳（微秒） -->
</composite>
```

**总大小：6 (SOFH) + 20 (Header) = 26 字节**

**关键差异：**
- FIX SBE 有 SOFH 用于 TCP 流上的消息帧定界（HTTP/WebSocket 不需要）
- FIX SBE header 包含 `seqNum` 和 `sendingTime`，因为 FIX 协议要求消息有序且可追踪
- REST/WS API 不需要这些，因为 HTTP 本身有请求-响应配对，WebSocket 有帧边界

## 四、类型系统对比

### 4.1 基础数值类型

两套 schema 的十进制浮点数编码方式相同：mantissa（int64）+ exponent（int8）。

```xml
<!-- 两套 schema 都有 -->
<type name="mantissa64" primitiveType="int64"/>
<type name="exponent8"  primitiveType="int8"/>
```

但 `spot_3_3.xml` 额外定义了 128 位 mantissa（用于大数值如 volume）：

```xml
<!-- 仅 spot_3_3.xml -->
<type name="mantissa128" length="16" primitiveType="uint8"/>
```

`spot-fixsbe-1_1.xml` 没有 `mantissa128`，因为 FIX 消息中不需要传输累计成交量等大数值字段。

### 4.2 变长字符串类型

| 类型 | `spot_3_3.xml` | `spot-fixsbe-1_1.xml` | 说明 |
|------|---------------|----------------------|------|
| `varString` | ✅ | ✅ | length: uint16 |
| `varString8` | ✅ | ✅ | length: uint8 |
| `optionalVarString` | ✅ | ✅ | 空串表示 null |
| `optionalVarString8` | ✅ | ✅ | 空串表示 null |

### 4.3 消息内嵌类型

`spot_3_3.xml` 有独特的 **消息嵌套** 类型，用于在响应中嵌入子消息：

```xml
<!-- 仅 spot_3_3.xml — 用于 WebSocket API 的响应包装 -->
<composite name="messageData">
    <type name="length" primitiveType="uint32" maxValue="2147483647"/>
    <type name="varData" length="0" primitiveType="uint8"/>
</composite>
<composite name="messageData16">...</composite>
<composite name="messageData8">...</composite>
<composite name="optionalMessageData">...</composite>
<composite name="optionalMessageData16">...</composite>
```

这些类型用于：
- `WebSocketResponse.result` — 包装实际的业务响应消息
- `ExchangeInfoResponse.symbols[].filters[].filter` — 嵌入不同类型的过滤器消息
- `ErrorResponse.data` — 嵌入附加错误数据

`spot-fixsbe-1_1.xml` **没有这些嵌套类型**，因为 FIX 协议中每条消息是独立的，不需要包装。

### 4.4 Repeating Group 维度类型

| 类型 | `spot_3_3.xml` | `spot-fixsbe-1_1.xml` | 说明 |
|------|---------------|----------------------|------|
| `groupSizeEncoding` | ✅ (blockLength: u16, numInGroup: u32) | — | 大型重复组 |
| `groupSize32Encoding` | — | ✅ (blockLength: u16, numInGroup: u32) | FIX 版大型重复组 |
| `groupSize16Encoding` | ✅ (blockLength: u16, numInGroup: u16) | ✅ (blockLength: u16, numInGroup: u16) | 中型重复组 |
| `smallGroupSize16Encoding` | — | ✅ (blockLength: u8, numInGroup: u16) | FIX 版小型重复组 |
| `groupSize8Encoding` | — | ✅ (blockLength: u16, numInGroup: u8) | 极小重复组 |
| `smallGroupSize8Encoding` | — | ✅ (blockLength: u8, numInGroup: u8) | FIX 版最小重复组 |

`spot-fixsbe-1_1.xml` 有更多尺寸的组编码，是为了在 FIX 场景下精确控制 payload 大小。

## 五、枚举定义风格对比

这是两套 schema 差异最直观的地方。

### 5.1 `spot_3_3.xml` — 使用整数编码 + jsonValue 注释

```xml
<enum name="orderSide" encodingType="uint8">
    <validValue name="Buy"               mbx:jsonValue="BUY">0</validValue>
    <validValue name="Sell"              mbx:jsonValue="SELL">1</validValue>
    <validValue name="NonRepresentable"  mbx:jsonValue="NON_REPRESENTABLE">254</validValue>
</enum>

<enum name="orderType" encodingType="uint8">
    <validValue name="Market"            mbx:jsonValue="MARKET">0</validValue>
    <validValue name="Limit"             mbx:jsonValue="LIMIT">1</validValue>
    <validValue name="StopLoss"          mbx:jsonValue="STOP_LOSS">2</validValue>
    <validValue name="StopLossLimit"     mbx:jsonValue="STOP_LOSS_LIMIT">3</validValue>
    <validValue name="TakeProfit"        mbx:jsonValue="TAKE_PROFIT">4</validValue>
    <validValue name="TakeProfitLimit"   mbx:jsonValue="TAKE_PROFIT_LIMIT">5</validValue>
    <validValue name="LimitMaker"        mbx:jsonValue="LIMIT_MAKER">6</validValue>
    <validValue name="NonRepresentable"  mbx:jsonValue="NON_REPRESENTABLE">254</validValue>
</enum>
```

**特点：**
- 编码类型统一为 `uint8`
- 值为连续整数（0, 1, 2, ...）
- 有 `mbx:jsonValue` 属性标注对应的 JSON 字符串
- 所有 enum 都有 `NonRepresentable`（254）占位，用于前向兼容

### 5.2 `spot-fixsbe-1_1.xml` — 使用 FIX 标准字符编码

```xml
<enum name="side" encodingType="char">
    <validValue name="Buy">1</validValue>
    <validValue name="Sell">2</validValue>
    <validValue name="NonRepresentable">~</validValue>
</enum>

<enum name="ordType" encodingType="char">
    <validValue name="Market">1</validValue>
    <validValue name="Limit">2</validValue>
    <validValue name="Stop">3</validValue>
    <validValue name="StopLimit">4</validValue>
    <validValue name="Pegged">P</validValue>
    <validValue name="NonRepresentable">~</validValue>
</enum>
```

**特点：**
- 编码类型大量使用 `char`（单字节 ASCII 字符）
- 值沿用 FIX 协议标准 tag 值（如 Side: 1=Buy, 2=Sell）
- `NonRepresentable` 使用 `~` 字符（0x7E）
- 没有 `mbx:jsonValue`（FIX 不需要对应 JSON）
- 字段名使用 FIX 标准名称（如 `ordType` 而非 `orderType`，`ordStatus` 而非 `orderStatus`）

### 5.3 同一概念在两套 schema 中的枚举值对比

| 概念 | `spot_3_3.xml` 名称 / 编码类型 / 值 | `spot-fixsbe-1_1.xml` 名称 / 编码类型 / 值 |
|------|------|------|
| 买入 | `orderSide.Buy` / uint8 / `0` | `side.Buy` / char / `'1'` |
| 卖出 | `orderSide.Sell` / uint8 / `1` | `side.Sell` / char / `'2'` |
| 市价单 | `orderType.Market` / uint8 / `0` | `ordType.Market` / char / `'1'` |
| 限价单 | `orderType.Limit` / uint8 / `1` | `ordType.Limit` / char / `'2'` |
| GTC | `timeInForce.Gtc` / uint8 / `0` | `timeInForce.GoodTillCancel` / char / `'1'` |
| IOC | `timeInForce.Ioc` / uint8 / `1` | `timeInForce.ImmediateOrCancel` / char / `'3'` |
| FOK | `timeInForce.Fok` / uint8 / `2` | `timeInForce.FillOrKill` / char / `'4'` |
| 新订单 | `orderStatus.New` / uint8 / `0` | `ordStatus.New` / char / `'0'` |
| 已取消 | `orderStatus.Canceled` / uint8 / `3` | `ordStatus.Canceled` / char / `'4'` |
| 已拒绝 | `orderStatus.Rejected` / uint8 / `5` | `ordStatus.Rejected` / char / `'8'` |

> **绝对不能将一套 schema 的枚举值用于另一套**——相同概念的编码完全不同。

## 九、如何选择

| 场景 | 推荐使用 | 理由 |
|------|---------|------|
| **一般量化策略开发** | `spot_3_3.xml` (REST/WS API) | API 功能全面，接入简单，支持查询+交易+行情 |
| **超低延迟交易（机构级）** | `spot-fixsbe-1_1.xml` (FIX API) | FIX 协议延迟更低，支持推送模式的 ExecutionReport |
| **超低延迟行情获取** | `stream_1_0.xml` (SBE Market Data Streams) | 专用行情数据流，payload 最小，延迟最低 |
| **同时需要低延迟交易+行情** | FIX OE (`spot-fixsbe-1_1.xml`) + FIX MD (`spot-fixsbe-1_1.xml`) | 一套 schema，两个会话 |
| **仅需要行情数据** | SBE Market Data Streams (`stream_1_0.xml`) 或 FIX MD | 看是否已有 FIX 基础设施 |
| **仅需要 Drop Copy（交易回报旁路）** | `spot-fixsbe-1_1.xml` + FIX DC 会话 | Drop Copy 仅 FIX 支持 |

## 十、版本废弃策略

两套 schema 共享相同的废弃策略（来自 [SBE FAQ](https://developers.binance.com/docs/zh-CN/binance-spot-api-docs/faqs/sbe_faq)）：

1. 新版本发布时，旧版本立即被标记为 **废弃（deprecated）**
2. 废弃后**至少 6 个月**仍可使用
3. 6 个月后可能被 **停用（retired）**
4. 使用废弃 schema 时会收到提示：
   - REST API：响应包含 `X-MBX-SBE-DEPRECATED` header
   - WebSocket API：`sbeSchemaIdVersionDeprecated = true`
   - FIX API：`LogonAck` 中 `sbeSchemaIdVersionDeprecated = true`

**当前生命周期状态（截至 2026-03-11）：**

| Schema | 最新版 | 废弃版 | 已停用版 |
|--------|-------|-------|---------|
| REST/WS API | 3:3 | 3:2（2026-03-09 废弃） | 3:0（2026-02-19 停用）, 2:1, 2:0, 1:0 |
| FIX API | 1:1 | 1:0（2026-03-09 废弃） | — |

生命周期 JSON 文件位置：
- REST/WS: [`sbe_schema_lifecycle_prod.json`](https://github.com/binance/binance-spot-api-docs/blob/master/sbe/schemas/sbe_schema_lifecycle_prod.json)
- FIX: [`sbe_fix_schema_lifecycle_prod.json`](https://github.com/binance/binance-spot-api-docs/blob/master/sbe/schemas/sbe_fix_schema_lifecycle_prod.json)

## 十一、Payload 结构对比示例

以"新建限价买单"为例，展示两种 schema 的消息结构差异。

### 11.1 `spot_3_3.xml` — 新建订单的响应 (`NewOrderResultResponse`, ID=301)

```
[messageHeader: 8 bytes]
  blockLength | templateId=301 | schemaId=3 | version=3

[fixed fields]
  priceExponent:  int8
  qtyExponent:    int8
  orderId:        int64
  orderListId:    int64 (optional, -1 = null)
  transactTime:   int64 (微秒)
  price:          int64 (mantissa)
  origQty:        int64 (mantissa)
  executedQty:    int64 (mantissa)
  cummulativeQuoteQty: int64 (mantissa)
  status:         uint8 (orderStatus enum)
  timeInForce:    uint8
  orderType:      uint8
  side:           uint8 (orderSide enum)
  ... 更多可选字段 ...
  expiryReason:   uint8 (sinceVersion=3, optional)

[variable-length data]
  symbol:         varString8
  clientOrderId:  varString8
```

### 11.2 `spot-fixsbe-1_1.xml` — 执行报告 (`ExecutionReport`, ID=98)

```
[sofh: 6 bytes]
  messageLength | encodingType=0xEB50

[messageHeader: 20 bytes]
  blockLength | templateId=98 | schemaId=1 | version=1 | seqNum | sendingTime

[fixed fields]
  PriceExponent:  int8
  QtyExponent:    int8
  ExecID:         int64 (optional)
  OrderID:        int64 (optional)
  OrderQty:       int64 (mantissa, optional)
  OrdType:        char  ('1'=Market, '2'=Limit, ...)
  Side:           char  ('1'=Buy, '2'=Sell)
  ExecInst:       char  (optional)
  Price:          int64 (mantissa, optional)
  ... Trigger 字段 ...
  ... Peg 字段 ...
  TimeInForce:    char  ('1'=GTC, '3'=IOC, '4'=FOK)
  TransactTime:   int64 (微秒, optional)
  ExecType:       char  ('0'=New, '4'=Canceled, 'F'=Trade, ...)
  CumQty:         int64 (mantissa)
  OrdStatus:      char  ('0'=New, '1'=PartiallyFilled, ...)
  ... 更多可选字段 ...
  ExpiryReason:   uint8 (sinceVersion=1, optional)

[repeating group: MiscFees]
  blockLength | numInGroup (smallGroupSize16Encoding)
  forEach: Exponent, MiscFeeAmt, MiscFeeType, MiscFeeCurr

[variable-length data]
  ClOrdID, OrigClOrdID, Symbol, SecondarySymbol, CounterSymbol, ErrorText
```

**关键差异总结：**
1. FIX 版有额外的 26 字节 SOFH+header 开销
2. FIX 版的枚举使用 char 编码（如 Side='1'），REST/WS 版使用 uint8（如 side=0）
3. FIX 版字段名使用 PascalCase + FIX 标准缩写，REST/WS 版使用 camelCase
4. FIX 版的手续费通过 `MiscFees` 重复组传递，REST/WS 版通过 `fills` 组中的 `commission` 字段
5. FIX 版有独立的 `ExecType` 和 `OrdStatus` 字段，REST/WS 版在不同消息中分别定义
